---
title: Production Build
description: Bundle a Vercube + Vite app for production with vite build
---

`@vercube/vite` plugs into Vite's `buildApp` orchestration, so a single `vite build` produces a runnable server — and, if your project has a frontend, builds it too.

## Building

```bash
npx vite build
```

Any other environments (such as a frontend `client` built from `index.html`) are built first, then the Vercube server: the generated entry — together with your controllers, services, and the Vercube framework code in your source graph — is bundled into `dist/index.mjs`.

```
dist/
  index.mjs          # the Vercube server
  index.html         # the frontend (only if your project has one)
  assets/…           # frontend assets
```

Bare dependencies (npm packages and `@vercube/*`) are **externalized**: the output imports them from `node_modules` at runtime rather than inlining them, which keeps the bundle small and matches Vercube's existing build behaviour.

## Running

The built entry exports a `fetch` handler and starts an HTTP listener when run directly:

```bash
node dist/index.mjs
```

```ansi
➜ Listening on: http://[::1]:3000/
```

The port and host come from your Vercube configuration (`vercube.config.ts`) or its defaults, exactly as in a standard Vercube application.

### Using the fetch handler

Because the entry also exports `fetch`, you can mount it in any environment that speaks the Web `fetch` API (edge runtimes, serverless adapters, tests) instead of starting the built-in listener:

```ts
import { fetch } from './dist/index.mjs';

const response = await fetch(new Request('http://localhost/api/users'));
```

## Configuration

The build reads the same `vercube.config.ts` as the rest of the framework. The plugin writes its output to `dist/` next to your project root; adjust your deployment to ship `dist/` together with `node_modules` (or run your own bundling/externals step if you need a fully standalone artifact).
