---
title: Production Build
description: Bundle a Vercube + Vite app for production with vite build
---

`@vercube/vite` plugs into Vite's `buildApp` orchestration, so a single `vite build` produces a runnable server, and builds your frontend too when the project has one.

## Building

```bash
npx vite build
```

If the project has a frontend (an `index.html`), it is built first into `dist/public`, then the Vercube server is bundled into `dist/index.mjs`. The server bundle includes your controllers, services and the Vercube framework code from your source graph.

```
dist/
  index.mjs           # the Vercube server
  public/             # the frontend (only if the project has one)
    index.html
    assets/…
```

Bare dependencies (npm packages and `@vercube/*`) are externalized: the output imports them from `node_modules` at runtime rather than inlining them, which keeps the bundle small and matches Vercube's existing build behaviour.

## Running

```bash
node dist/index.mjs
```

```ansi
➜ Listening on: http://[::1]:3000/
```

This serves your API routes and, when a frontend was built, the static files from `dist/public` (with `index.html` for `/`). One process serves both, the same way the dev server does. The port and host come from your Vercube configuration (`vercube.config.ts`) or its defaults.

### Using the fetch handler

Because the entry also exports `fetch`, you can mount it in any environment that speaks the Web `fetch` API (edge runtimes, serverless adapters, tests) instead of starting the built-in listener:

```ts
import { fetch } from './dist/index.mjs';

const response = await fetch(new Request('http://localhost/api/users'));
```

## Configuration

The build reads the same `vercube.config.ts` as the rest of the framework. Output goes to `dist/` in your project root. Deploy `dist/` together with `node_modules`, or run your own bundling step if you need a fully standalone artifact.
