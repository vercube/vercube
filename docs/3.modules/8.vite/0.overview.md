The `@vercube/vite` package runs a Vercube server as a [Vite](https://vite.dev) plugin, built on Vite's [Environment API](https://vite.dev/guide/api-environment.html). You get Vercube's decorator-based routing and full dependency injection with Vite's dev server, fast server-side HMR, and a single build command — while keeping the integration **zero-config**: controllers and services are discovered automatically.

This is a standalone alternative to the [Nitro integration](/modules/nitro/overview). It does **not** replace the `vercube dev` / `vercube build` CLI — it is a regular Vite plugin you drive with `vite` and `vite build`.

## Installation

::code-group
  ```bash [pnpm]
  $ pnpm add -D @vercube/vite vite
  ```
  ```bash [npm]
  $ npm install -D @vercube/vite vite
  ```
  ```bash [bun]
  $ bun add -D @vercube/vite vite
  ```
::

### Requirements

- **Vite 8**
- `experimentalDecorators: true` must be set in your `tsconfig.json`

```json [tsconfig.json]
{
  "compilerOptions": {
    "experimentalDecorators": true,
    "emitDecoratorMetadata": false
  }
}
```

::callout{type="info" icon="i-lucide-info"}
Unlike the Nitro integration, Vercube's DI does **not** rely on `emitDecoratorMetadata` here — explicit `@Inject(Token)` is used throughout, so you can leave it off.
::

## Quick Start

::steps
### Configure Vite

Add the `vercube` plugin to your `vite.config.ts`:

```ts [vite.config.ts]
import { vercube } from '@vercube/vite';
import { defineConfig } from 'vite';

export default defineConfig({
  plugins: [vercube()],
  server: {
    port: 3000,
  },
});
```

### Create a Controller

Place a `@Controller` class anywhere under `src/` — it is discovered automatically:

```ts [src/Controllers/UserController.ts]
import { Controller, Get, Param } from '@vercube/core';
import { Inject } from '@vercube/di';
import { UserService } from '../Services/UserService';

@Controller('/api/users')
export default class UserController {

  @Inject(UserService)
  private userService: UserService;

  @Get('/')
  async list() {
    return this.userService.findAll();
  }

  @Get('/:id')
  async get(@Param('id') id: string) {
    return this.userService.findById(id);
  }
}
```

### Create a Service

::callout{type="warning" icon="i-lucide-triangle-alert"}
As with the Nitro integration, auto-discovery uses `@Injectable()` as an explicit opt-in signal. Decorate every service you want auto-registered.
::

```ts [src/Services/UserService.ts]
import { Injectable } from '@vercube/di';

@Injectable()
export class UserService {
  findAll() {
    return [{ id: '1', name: 'Alice' }];
  }

  findById(id: string) {
    return { id, name: 'Alice' };
  }
}
```

### Run

```bash
npx vite
```

Edit or add controllers while the server runs — changes are picked up live, no restart needed.
::

## How It Works

The plugin defines a dedicated Vite **environment** named `vercube` and, at startup, scans your source tree with AST parsing ([`@vercube/scan`](https://github.com/vercube/vercube/tree/main/packages/scan)) to discover `@Controller` and `@Injectable` classes. From those it generates a server entry that:

1. Creates a Vercube application via `createApp()`
2. Binds all discovered controllers and services into the DI container
3. Flushes the container queue — which registers routes via decorator initialization
4. Exports the app's `fetch` handler

In **dev**, that entry runs inside an isolated [`env-runner`](https://www.npmjs.com/package/env-runner) worker through Vite's [`ModuleRunner`](https://vite.dev/guide/api-environment-runtimes.html#modulerunner). Vercube only claims **the routes you actually define** — a request matching a discovered route is converted (via `srvx/node`) and dispatched into the worker; everything else falls through to Vite. Adding or editing a controller reloads the worker, so the change is live.

The generated entry is written to `node_modules/.vercube/server-entry.mjs` (already ignored by git via `node_modules`).

## Using with a frontend

Because Vercube only handles its own routes, the plugin coexists with a frontend that Vite serves on the same dev server — add your framework's plugin alongside `vercube()`:

```ts [vite.config.ts]
import { vercube } from '@vercube/vite';
import vue from '@vitejs/plugin-vue';
import { defineConfig } from 'vite';

export default defineConfig({
  plugins: [
    vue(),
    vercube({ scanDirs: ['src/server'], setupFile: './src/server/Boot/Setup.ts' }),
  ],
});
```

Vite serves `index.html`, your components and HMR; requests to your `@Controller` routes (e.g. `/api/*`) go to Vercube. `vite build` builds the frontend and the server side by side. A full Vite + Vue example lives in [`examples/vite`](https://github.com/vercube/vercube/tree/main/examples/vite).

## Module Options

| Option | Type | Default | Description |
|---|---|---|---|
| `rootDir` | `string` | Vite `root` | Project root used to resolve `scanDirs` and `setupFile`. |
| `scanDirs` | `string[]` | `['src']` | Directories scanned recursively for `@Controller` and `@Injectable` classes. |
| `setupFile` | `string` | `undefined` | Module whose default export `(app: App) => void \| Promise<void>` runs as `createApp`'s setup hook — **before** the app initializes. See [Customizing the Container](#customizing-the-container). |
| `runner` | `string` | `'node-worker'` | The `env-runner` runner used to execute server code in dev. |

```ts [vite.config.ts]
vercube({
  scanDirs: ['src', 'modules'],
  setupFile: './src/Boot/Setup.ts',
})
```

## Customizing the Container

Auto-discovery binds `@Controller` and `@Injectable` classes. Anything it cannot infer — registering plugins, mounting storage, or binding an interface token to an implementation — belongs in a `setupFile`.

The file is passed as `createApp`'s `setup` hook, so it runs **before** the app initializes. That timing matters: plugin registration (`app.addPlugin`) must happen before initialization.

```ts [vite.config.ts]
import { vercube } from '@vercube/vite';
import { defineConfig } from 'vite';

export default defineConfig({
  plugins: [vercube({ setupFile: './src/Boot/Setup.ts' })],
});
```

```ts [src/Boot/Setup.ts]
import type { App } from '@vercube/core';
import { AuthProvider } from '@vercube/auth';
import { StorageManager } from '@vercube/storage';
import { MemoryStorage } from '@vercube/storage/drivers/MemoryStorage';
import { BasicAuthenticationProvider } from '../Services/BasicAuthenticationProvider';

export default async function setup(app: App) {
  // bind an interface token to an implementation
  app.container.bind(AuthProvider, BasicAuthenticationProvider);

  // mount storage
  app.container.bind(StorageManager);
  app.container.get(StorageManager).mount({ storage: MemoryStorage });
}
```

## WebSockets

The [`@vercube/ws`](/modules/web-sockets/overview) plugin works in both the dev server and the production build. Register it from your `setupFile` and define WebSocket controllers (`@Controller` + `@Namespace`) as usual — they are auto-discovered alongside HTTP controllers.

```ts [src/Boot/Setup.ts]
import { WebsocketPlugin } from '@vercube/ws';
import type { App } from '@vercube/core';

export default async function setup(app: App) {
  app.addPlugin(WebsocketPlugin);
}
```

The dev server forwards WebSocket upgrade handshakes into the worker; the production build handles them natively through `srvx`.
